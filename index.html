String Catalog调研

调研 String Catalog 的 iOS 国际化实践总结

基于 https://developer.apple.com/documentation/xcode/localizing-and-varying-text-with-a-string-catalog

一、为什么要使用 String Catalog（.xcstrings）

在 Xcode 15 及之后，Apple 推出了 String Catalog，作为对传统 Localizable.strings 的升级方案，核心目标是：

  • 集中管理所有可本地化字符串
  • 自动提取代码、Storyboard、XIB 中的字符串
  • 原生支持 复数、性别、设备变体 等复杂场景
  • 与 SwiftUI / UIKit 深度集成

相比传统 .strings 文件：

维度 Localizable.strings String Catalog
自动提取 ❌ ✅
复数/变体 手写 结构化支持
UI 编辑 文本 图形化编辑
SwiftUI 支持 一般 优秀
翻译进度可视化 ❌ 无法直观看到 ✅ 可查看每种语言完成度
翻译状态标记 ❌ 无 ✅ 已翻译打勾 / Need Review 提示

二、String Catalog 的核心概念

1. .xcstrings 是什么

.xcstrings 是一种 结构化的字符串资源文件，Xcode 会：

  • 扫描项目源码
  • 收集可本地化字符串
  • 自动写入 String Catalog

开发者只需要关注 声明字符串，而不是维护文件格式。

2. 自动提取机制

Xcode 会自动提取以下来源中的字符串：

  • NSLocalizedString("key", comment: "")
  • String(localized: "key")
  • SwiftUI 中的 Text("key")
  • Storyboard / XIB 中勾选了 Localize 的文本

⚠️ 前提条件：

  • 字符串必须是 字面量
  • 代码必须属于 当前 Target 的编译单元（这个问题导致了pod库里代码文件不能自动提取）

三、String Catalog 与 Swift 的推荐用法

1. Swift 原生 API

优势：

  • 编译期检查
  • 自动关联 String Catalog
  • 更符合 Swift Concurrency / SwiftUI 设计

2. 传统 NSLocalizedString

依然支持，支持iOS15以下的版本。

四、变体与复数

String Catalog 的一大亮点是 内建语义结构。

1. 复数

例如：

  • 1 file
  • 2 files

在 .xcstrings 中可以直接配置：

  • zero
  • one
  • other

无需再手写 .stringsdict。

2. 设备 / 上下文变体

支持根据：

  • 语言
  • 地区
  • 设备（iPhone / iPad）

配置不同文案，Xcode 会自动选择最合适的字符串。

五、重点问题：为什么 Pod 库源码不会自动提取到 String Catalog？

1. 根本原因

String Catalog 只会提取“当前 Target 源码”中的字符串。

而大多数 Pod 库：

  • 位于 Pods/ 目录
  • 不属于 App 主 Target 的编译单元

因此 不会被自动扫描进 App 的 .xcstrings 文件。

Pod / Framework 通常：

  • 自带 Localizable.strings
  • 使用 Bundle.module / Bundle(for:)
  • 在自己的 Bundle 内完成本地化

试图让 Pod 源码被自动提取

目前 不可行

  • Xcode 不支持跨 Target 自动提取
  • 强行 Copy Source 会破坏依赖结构

六、结语

String Catalog 代表了 Apple 对国际化的长期方向：

  • 更结构化
  • 更自动化
  • 更 Swift 化

ios缓存

概述

在iOS中缓存主要分为两种，一种是内存缓存一种是磁盘缓存。内存缓存提供容量小但是高速的存取功能，磁盘缓存提供容量大但是低速的存取功能。在使用的时候一般是将最近的数据（如一天）存储在内存缓存中；将超出最近时间而又在合适时间内的数据（如超过一天在一周内）从内存缓存中清除，将其存储在磁盘缓存中；将超出最大时间（如超过一周）的数据从磁盘中销毁。

内存缓存

UserDefaults

  • 用途：存储少量用户设置或配置。
  • 优点：简单易用。
  • 缺点：不适合存储大量数据。
    当你对 UserDefaults 进行修改后，系统会自动将这些改动写入到磁盘。这个写入过程是异步进行的，因此不会立即反映。如果需要立即将更改同步到磁盘，可以调用 synchronize() 方法，但这通常是不必要的，因为 iOS 系统会管理好这个过程。

NSCache

  • 用途：内存中缓存大量临时数据。
  • 优点：自动管理内存，防止应用占用过多内存。
  • 缺点：数据不会被持久化存储。
    它是线程安全并不retain key。它的底层直接调用了 libcache.dylib，通过pthead_mutex（互斥锁）完成了线程安全。但是由于它的性能和key值的相似度有关，如果有大量相似的key的话，NSCache的存取性能下降地厉害。

不同类锁的速度

磁盘缓存

  • 用途：持久化存储大量数据。
  • 方法：通过FileManager操作文件系统。
  • 优点：适合存储大数据量，如离线数据。
  • 缺点：读写速度慢于内存。

第三方库

  • 例如：Kingfisher（图片缓存），AlamofireImage。
  • 优点：提供高级功能，如图片解码和内存管理。
  • 缺点：增加应用依赖。

缓存策略

  • 选择依据：数据的大小、更新频率和用户访问模式。
  • 时效性与一致性：确定数据需要多久更新一次，以及在应用中需要保持多高的数据一致性。
  • 资源利用平衡：根据应用的具体需求和设备能力，平衡内存与磁盘的使用，确保应用运行流畅不占用过多系统资源。

协议

  • 默认情况下，协议中定义的内容必须全部都实现
  • 协议中定义属性必须用var关键字
  • 只有将协议中的实例方法标记为mutating，才允许结构体、枚举的具体实现修改自身内存，类在实现方法时不用加mutating，结构体、枚举才需要加mutating
  • 协议中还可以定义初始化器init，非final类实现时必须加上required，目的是为了让所有遵守这个协议的类都拥有初始化器，所以加上required强制子类必须实现，除非是加上final不需要子类的类
