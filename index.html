ios缓存

概述

在iOS中缓存主要分为两种，一种是内存缓存一种是磁盘缓存。内存缓存提供容量小但是高速的存取功能，磁盘缓存提供容量大但是低速的存取功能。在使用的时候一般是将最近的数据（如一天）存储在内存缓存中；将超出最近时间而又在合适时间内的数据（如超过一天在一周内）从内存缓存中清除，将其存储在磁盘缓存中；将超出最大时间（如超过一周）的数据从磁盘中销毁。

内存缓存

UserDefaults

  • 用途：存储少量用户设置或配置。
  • 优点：简单易用。
  • 缺点：不适合存储大量数据。
    当你对 UserDefaults 进行修改后，系统会自动将这些改动写入到磁盘。这个写入过程是异步进行的，因此不会立即反映。如果需要立即将更改同步到磁盘，可以调用 synchronize() 方法，但这通常是不必要的，因为 iOS 系统会管理好这个过程。

NSCache

  • 用途：内存中缓存大量临时数据。
  • 优点：自动管理内存，防止应用占用过多内存。
  • 缺点：数据不会被持久化存储。
    它是线程安全并不retain key。它的底层直接调用了 libcache.dylib，通过pthead_mutex（互斥锁）完成了线程安全。但是由于它的性能和key值的相似度有关，如果有大量相似的key的话，NSCache的存取性能下降地厉害。

不同类锁的速度

磁盘缓存

  • 用途：持久化存储大量数据。
  • 方法：通过FileManager操作文件系统。
  • 优点：适合存储大数据量，如离线数据。
  • 缺点：读写速度慢于内存。

第三方库

  • 例如：Kingfisher（图片缓存），AlamofireImage。
  • 优点：提供高级功能，如图片解码和内存管理。
  • 缺点：增加应用依赖。

缓存策略

  • 选择依据：数据的大小、更新频率和用户访问模式。
  • 时效性与一致性：确定数据需要多久更新一次，以及在应用中需要保持多高的数据一致性。
  • 资源利用平衡：根据应用的具体需求和设备能力，平衡内存与磁盘的使用，确保应用运行流畅不占用过多系统资源。

调用时出现的问题

由于Objective-C和Swift中类型不完全一样。使得在swift定义的model可能无法被Oc接口使用。
例如Objective-C中nil 可以用于任何对象。在swift文件中只有可选类型才能被设为nil。
swift的枚举和sturct都需要特别标记才能在Objective-C中使用

如果在Oc中使用将会失败

  • 1.结构体问题：Objective-C 不能直接创建或操作 Swift 的 struct，因为 Objective-C 仅支持类。
  • 2.枚举问题：尽管 Swift 的枚举可以通过 @objc 显式地映射到 Objective-C，但这仅限于整型值的枚举。在上面的 Swift 代码中，VehicleType 使用了字符串作为原始值，这在 Objective-C 中无法直接使用。
  • 3.泛型和复杂集合：如果 Swift 类使用了泛型或包含 Swift 特定集合（如 Array），这些在 Objective-C 中也不能直接使用。
    解决方案：用 NSObject 的子类来替换 Swift 的 struct。确保枚举是基于整型，并通过 @objc 标记。避免使用 Swift 特定的泛型集合或在 Objective-C 中不支持的类型。

##先定义TargetType协议的枚举来配置网络请求

  • TargetType 是 Moya 的核心，每个网络请求都基于它来配置。它要求定义几个基本但关键的属性：
  • baseURL （基础URL）：所有请求的基本URL。
  • path （路径）：与 baseURL 结合，形成完整的请求URL。
  • method （方法）：HTTP请求的方法，如 GET、POST 等。
  • task （任务）：定义请求的具体内容，如是否有参数，参数是怎样编码的，是否上传文件等。
  • task 是 TargetType 中最复杂的部分，它定义了发送请求时的具体行为和数据。task 有几种类型，每种类型适用于不同的网络请求场景：
    – requestPlain：发送一个基本的请求，不带任何参数。
    – requestData(Data)：允许你发送一些二进制数据作为请求体。
    – requestParameters(parameters: [String: Any], encoding: ParameterEncoding)：使用这种类型可以发送带参数的请求。参数可以是 .url、.json、.formData 等格式编码。
    – requestCompositeData(bodyData: Data, urlParameters: [String: Any])：允许在同一个请求中同时发送 URL 参数和请求体数据。
    – uploadMultipart([MultipartFormData])：用于上传文件或数据时，可以构建复杂的多部分请求体。
  • headers （头部）：为请求指定 HTTP 头部。 Moya 返回的 Response 对象包含了完整的 HTTP 响应数据。你可以使用 Moya 的扩展方法来简化数据处理，如：
  • response.mapJSON(): 将响应数据映射为 JSON 对象。
  • response.map(): 将响应数据映射为指定的数据模型，通常需要遵循 Decodable 协议
    使用Moya的provider来发送请求： 使用插件

##lazy关键词
在Swift中，lazy关键字用于延迟初始化一个属性直到它第一次被调用或访问。这意味着，只有在实际需要使用这个属性的时候，它才会被创建和初始化。这种方式通常用于初始化成本较高的对象，或者在对象的构造过程中依赖于某些条件或其他只有运行时才能确定的数据。
使用lazy可以让程序的启动更快，因为它避免了在应用启动时不必要的初始化工作。此外，它还能确保属性只在真正需要时才被创建，从而节省资源。

闭包和函数

定义

  • 函数：函数是具有名称的代码块，可以执行特定的任务。函数可以接收参数、执行操作，并返回结果。函数通常定义在全局作用域或类/结构体的上下文中。
  • 闭包：闭包是可以捕获和存储其所在上下文中的任何常量和变量引用的匿名函数。闭包通常以匿名的形式书写，它可以捕获其定义环境中的值。

捕获上下文

  • 函数：普通函数不能捕获或存储它们所在作用域之外的变量。
  • 闭包：闭包能够捕获周围作用域中的常量和变量，并持有这些值，即使作用域本身已经不存在了，闭包依然可以访问和修改这些值。

书写形式

  • 函数：通常定义得较为明确，具有固定的名称和结构（例如 func doSomething() {}）。
  • 闭包：书写更为灵活，通常用于短小的代码片段，可以直接内嵌于函数调用参数中或者用于变量赋值。闭包的语法结构通常更为紧凑（例如 { (parameters) -> return type in statements }）。

使用场景

  • 函数：更适合执行复杂或多步骤的操作，它们的重用性和结构性更强。
  • 闭包：非常适合用作完成特定任务的代码块，特别是在需要操作函数类型的值时，如回调函数、函数作为参数等场景。 这种方式的好处是代码的灵活性和封装性。不需要为每种行为定义一个单独的函数，也不需要在一个大的事件处理器中使用条件语句来区分行为。每个按钮的行为都是独立定义的，且可以根据需要随时更改，使得代码更加模块化和易于维护。

逃逸闭包

在 Swift 中，当一个闭包作为参数传递给一个函数，并且这个闭包在函数返回之后才被调用，我们称这个闭包为逃逸闭包。换句话说，逃逸闭包是在函数执行完毕后仍然可以在函数外部被调用的闭包。
逃逸闭包主要用于异步操作，比如网络请求、定时器等。在这些情况下，我们需要在函数执行完毕后继续使用这个闭包，比如在网络请求返回数据后更新 UI，或者在定时器触发后执行一些操作。

Person *p1 = [Person new];
Person *p1 会在栈内存中申请一个空间，在栈内存中声明一个Person类型的指针变量p1。p1是一个指针变量，那么只能存储地址。
[Person new]，真正在内存中创建对象的其实是这句代码
第一是在堆内存中申请1块合适大小的空间；
第二是在这个空间中根据类的模板创建对象，类模板中定义了什么属性，就把这些属性依次的声明在对象之中，对象中还有另外一个属性，叫做isa，它是一个指针，这个指针指向这个对象所属的类在代码段中的地址；
第三是初始化对象的属性。如果属性的类型是基本数据类型，那么赋值为0，如果属性的类型是OC的类指针类型，那么就赋值为nil；
第四是返回对象的地址，将这个地址赋值给p1。
因此对象的实际数据（即 Person 对象本身）是存储在堆上的，而指向这些数据的指针（即 p1）是存储在栈上的。

为什么在 Objective-C 中，所有的对象都用一个指针来追踪

  • 很多时候，我们需要把一个对象交给另一个函数或者方法去修改其中的内容（比如说一个 Person 对象的 age 属性），显然如果我们想让修改方获取到这个对象，我们需要的传递的是地址，而不是复制一份。
  • 另一方面，对象的内存是分配在堆上的，而值类型是分配到栈上的。所以一般对象的生命周期会比普通的值类型要长，而且创建和销毁对象以及内存管理是要消耗性能的，所以通过指针来引用一个对象，比直接复制和创建对象要更有效率、更节省性能。
