ios缓存

概述

在iOS中缓存主要分为两种，一种是内存缓存一种是磁盘缓存。内存缓存提供容量小但是高速的存取功能，磁盘缓存提供容量大但是低速的存取功能。在使用的时候一般是将最近的数据（如一天）存储在内存缓存中；将超出最近时间而又在合适时间内的数据（如超过一天在一周内）从内存缓存中清除，将其存储在磁盘缓存中；将超出最大时间（如超过一周）的数据从磁盘中销毁。

内存缓存

UserDefaults

  • 用途：存储少量用户设置或配置。
  • 优点：简单易用。
  • 缺点：不适合存储大量数据。
    当你对 UserDefaults 进行修改后，系统会自动将这些改动写入到磁盘。这个写入过程是异步进行的，因此不会立即反映。如果需要立即将更改同步到磁盘，可以调用 synchronize() 方法，但这通常是不必要的，因为 iOS 系统会管理好这个过程。

NSCache

  • 用途：内存中缓存大量临时数据。
  • 优点：自动管理内存，防止应用占用过多内存。
  • 缺点：数据不会被持久化存储。
    它是线程安全并不retain key。它的底层直接调用了 libcache.dylib，通过pthead_mutex（互斥锁）完成了线程安全。但是由于它的性能和key值的相似度有关，如果有大量相似的key的话，NSCache的存取性能下降地厉害。

不同类锁的速度

磁盘缓存

  • 用途：持久化存储大量数据。
  • 方法：通过FileManager操作文件系统。
  • 优点：适合存储大数据量，如离线数据。
  • 缺点：读写速度慢于内存。

第三方库

  • 例如：Kingfisher（图片缓存），AlamofireImage。
  • 优点：提供高级功能，如图片解码和内存管理。
  • 缺点：增加应用依赖。

缓存策略

  • 选择依据：数据的大小、更新频率和用户访问模式。
  • 时效性与一致性：确定数据需要多久更新一次，以及在应用中需要保持多高的数据一致性。
  • 资源利用平衡：根据应用的具体需求和设备能力，平衡内存与磁盘的使用，确保应用运行流畅不占用过多系统资源。

混合开发

桥接头文件配置

当在swift 项目中.创建一个objective-c的类 会有提示 选择 Create Bridging Header
Bridging Header 文件中导入所有的需要使用的 Objective-C 类的头文件
通过#import “x” 使得在swift文件中可以调用oc文件的接口

调用时出现的问题

由于Objective-C和Swift中类型不完全一样。使得在swift定义的model可能无法被Oc接口使用。
例如Objective-C中nil 可以用于任何对象。在swift文件中只有可选类型才能被设为nil。
swift的枚举和sturct都需要特别标记才能在Objective-C中使用

如果在Oc中使用将会失败

  • 1.结构体问题：Objective-C 不能直接创建或操作 Swift 的 struct，因为 Objective-C 仅支持类。
  • 2.枚举问题：尽管 Swift 的枚举可以通过 @objc 显式地映射到 Objective-C，但这仅限于整型值的枚举。在上面的 Swift 代码中，VehicleType 使用了字符串作为原始值，这在 Objective-C 中无法直接使用。
  • 3.泛型和复杂集合：如果 Swift 类使用了泛型或包含 Swift 特定集合（如 Array），这些在 Objective-C 中也不能直接使用。
    解决方案：用 NSObject 的子类来替换 Swift 的 struct。确保枚举是基于整型，并通过 @objc 标记。避免使用 Swift 特定的泛型集合或在 Objective-C 中不支持的类型。
